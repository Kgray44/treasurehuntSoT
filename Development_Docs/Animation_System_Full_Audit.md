# Animation System Full Audit

Status: complete audit of exact committed baseline `4dbe8c0ae2fbab2785d1d3f26b8d7ba33bf56aee`, with a dated Project Lanternwake Phase 1 working-tree reconciliation below. The original audit findings remain baseline evidence; the addendum identifies only the Phase 1 claims revalidated against current implementation.

## 1. Executive summary

The baseline has a substantial animation framework, but it does not yet have animation truth. The audit inventories **220** current effects, runtime contracts, and missing opportunities. Of the 28 registered GSAP scenes, **20** have a production call path and **8** are development-showcase-only, dead, or redundant. All 28 can build and finish without a required visible target. Therefore a completed timeline, a green builder test, and a clean showcase run do not prove that a user saw the intended object.

The most serious chain is in Player progression: root-scoped scenes query whichever section happens to be mounted; generic part names can match permanent content, a temporary event overlay, PageFlip sources, or PageFlip clones; presentation errors are caught; the snapshot refresh and viewed acknowledgment still continue. A mandatory event can consequently be marked viewed even when no required content was visible. Chapter release also keeps replay only in `lastRelease` component state, so refresh loses it. The live baseline run did render seal, parchment, and ink stages and exposed Replay Ceremony, but measured **10,081 ms**, failing the existing `<10,000 ms` acceptance limit.

All four named production Rive contracts lack a `.riv` path: mounted Invitation Seal and Finale Mechanism render static fallbacks, Journal Clasp and Voyage Compass are unmounted, and the only `.riv` is a development sample. Modern sign-in, Invitation Ceremony, Library, waiting-room, and Studio flows largely bypass the legacy cinematic paths and lack a unified Motion policy. Reduced motion has three competing authorities: custom product/system resolution, `MotionConfig reducedMotion="never"`, and CSS media queries.

## 2. Audit baseline and methodology

- Authoritative application snapshot: `4dbe8c0ae2fbab2785d1d3f26b8d7ba33bf56aee` in a detached worktree under `artifacts/validation/animation-audit/baseline`. Its tracked source is clean at that commit; two untracked audit database backups remain (`prisma/dev.db.pre-clean-20260718-125935` and `prisma/dev.db.rate-limit-polluted-20260718-125728`).
- The main checkout advanced only through synchronization commits during the audit and contained substantial concurrent uncommitted work. That work was treated as out of baseline and left untouched.
- Static work covered registry definitions, callers, triggers, selectors, `data-scene-part`, `data-gsap-owned`, `data-motion-owned`, `data-journal-actor`, `data-journal-phase`, state machines, library wrappers, assets, CSS, audio, event mapping, replay and acknowledgment paths.
- Dynamic work used an isolated port 3100 only after verifying port ownership; port 3000 was never reused or stopped. Focused tests and live target measurements were retained from the original audit task.
- Truth rule: **build/run/complete is not visible success**. Runtime proof requires an expected trigger, correct visible instance, required target count, uniqueness, stable final semantic state, and correct persistence/acknowledgment behavior.
- One final mutation flow was stopped when Next.js root inference proved the detached server was using the main checkout database. Only exact rows created by the audit run were transactionally removed after a backup and safety assertions. No further mutation run was attempted without a provably isolated database path.

## 3. Current architecture

`AnimationProvider` exposes product mode, `AnimationDirector`, audio, metrics, and asset/runtime state. The director resolves one of 28 registry scenes and gives the builder a root-scoped selector helper. Scene builders create GSAP timelines and semantic labels; Player and Quartermaster call them from event/action handlers. Motion independently owns React presence/layout. StPageFlip clones hidden React page sources into a separate visible host. Rive and Lottie wrappers mount asset contracts. CSS data attributes and a Web Animations observer drive the journal-opening phase chain.

The layering choices are sound in principle: GSAP for narrative sequencing, Motion for UI presence/layout, StPageFlip for physical curl, Rive for stateful authored objects, Lottie for decorative loops/one-shots, CSS for materials/ambient states, WAAPI for observing CSS completion, and Web Audio for semantic cues. The defect is coordination: target identity, ownership, reduced motion, interruption, final state, replay, and event acknowledgment are not governed by one enforceable contract.

## 6. Animations that do not run

The eight scenes above run only in the development showcase. The four production Rive objects do not run as Rive at all: Invitation Seal and Finale Mechanism mount fallbacks, while Journal Clasp and Voyage Compass are contract-only. Modern `PlayerSignIn`, `StaffSignIn`, and `InvitationCeremony` do not call the legacy `player-access`/`quartermaster-login` cinematic paths. The matrix's 18 broken items include these absent production paths plus infrastructure that makes successful completion unsafe.

## 7. Animations that run invisibly

- `parts()` returns zero-to-many nodes and no scene declares required, optional, unique, visible, or instance-scoped targets. All 28 timelines can complete with zero targets.
- Player event scenes query the Player root while only one section is mounted. Chapter content is absent outside Journal; Chart, Altar, Ledger, Log and Finale objects are likewise section-dependent.
- The temporary event overlay repeats generic part names also used by permanent sections. Static source inventory found 167 scene-part attributes but only 60 unique names. Examples: `route-path`, `map-fog`, and `artifact-reveal` each occur five times; several connection/light/stamp targets occur four times.
- Live landing evidence found `stars` at zero height and `arrival-copy` twice. Live journal evidence found `peripheral` twice and both visible.
- StPageFlip retains scene/ownership attributes in hidden React sources and cloned visible content. The tested page snapshot did not contain event-scene targets, so actual event-target duplication through PageFlip remains a structural risk, not a claimed observed occurrence.

## 8. Animations that run poorly

- Chapter release rendered but took 10,081 ms, over its `<10,000 ms` test contract.
- A wrong Quartermaster credential produced HTTP 401 but no `.form-error` within 10 seconds in the clean baseline acceptance retry.
- Mode changes recreate every Lottie runtime and can restart the non-looping ink bloom.
- Full-to-gentle does not update StPageFlip's 1100 ms timing to the intended 620 ms without remount.
- Director cleanup always reverts the GSAP context; route-changing success can briefly snap to its pre-scene state unless the route commits first. This is a code-path risk that still needs a visual checkpoint.
- Broad root queries animate duplicate nodes and do extra work. Blur/fog layers and repeated SVG path work are unbudgeted on mobile.

## 9. One-shot and replay analysis

Correct one-shot policies are first-arrival within a tab, an authoritative invitation acceptance, a server event's automatic ceremony, a destructive-command receipt, and a newly published immutable version. These should not mutate server state when replayed.

Incorrect policies are state stored only in component memory/session storage when the presentation is historically meaningful. Player progression events need a persisted event-history presentation payload plus a local `presentation status` (`unseen`, `presenting`, `presented`, `skipped`, `failed`, `replay`). Replay must be available from every relevant section, must use the original payload, must not send the command again, must restore the user's section/focus, and must work after refresh. `lastRelease` fails those requirements. Lottie mode recreation is accidental replay, not a replay policy.

## 10. Ownership conflicts

- `claimRuntimeOwnership` rejection only warns and returns false; the director does not stop the GSAP query/timeline. Motion does not register in the WeakMap.
- `VoyageChart` markers combine Motion transform/layout with GSAP ownership on the same button.
- `ShipsLog` applies Motion layout and GSAP to the same `motion.li`.
- `ArtifactInspection` applies Motion `layoutId` and declares the same node GSAP-owned; the GSAP scene is dormant today but unsafe to connect directly.
- Companion header/navigation both expose `peripheral`; chapter release dims two visible targets.
- Quartermaster combines Motion UI and broad GSAP root ownership.
- PageFlip sources and clones retain selector attributes. The fix is not to ban a library: put each engine on a dedicated wrapper/child and enforce claims before playback.

## 11. Wrong-library findings

StPageFlip must exclusively own page curl, so manual/programmatic GSAP page-flip scenes should be removed. Motion should own dialogs, drawers, cards, tabs, filters, shared layout, route content and list updates. GSAP should own bounded narrative order and SVG/path choreography, not general UI presence. CSS should retain materials, hover/focus and low-cost ambience, while narrative ordering leaves CSS state chains. Rive should own the four named stateful objects after real assets exist. Lottie should remain decorative and receive semantic segment/play commands for non-looping effects. WAAPI should observe CSS completion but not become an unbounded blocking state machine.

## 12. Lifecycle and cleanup findings

`waitForJournalPhase()` uses `Promise.allSettled(animation.finished)` without timeout or abort racing. Missing actor or missing animation resolves immediately; an infinite animation never resolves. Abort during that wait is ineffective until all promises settle. Director context reversion lacks a declared `commit final pose` versus `revert on failure` policy. Player event errors are caught and followed by acknowledgment. PageFlip recreates under some but not all setting changes. Every runtime needs an idempotent teardown, abort priority, final-state policy, focus handoff, and test for route unmount/mode change/visibility change.

## 13. Reduced-motion findings

`resolveMotionMode` correctly chooses the stricter product/system preference, but `MotionConfig reducedMotion="never"` prevents Motion from independently honoring the browser, and CSS still independently honors the media query. Components that do not call `useMotionMode` can diverge. Reduced showcase success proves the harness remains usable; it does not prove each product path produces the same semantic state. The target design is one resolved policy object consumed by GSAP durations/travel, Motion variants, PageFlip mode, Rive inputs, Lottie play/segments, CSS root attributes, and audio. Reduced mode must preserve order, meaning, focus and readable final state while eliminating spatial/continuous motion.

## 14. Performance findings

No production performance budget is enforced. Main risks are repeated root queries, duplicate target writes, Lottie recreation on every mode change, multiple fog runtimes, blur/filter animation, PageFlip clone recreation, idle CSS loops continuing while hidden, and possible listener/runtime leaks across route/replay cycles. The reduced showcase ended at approximately 161 FPS with three long tasks in one development run, but development telemetry and a reduced-only showcase are not representative production budgets. Use p95 frame time, long-task count, layout shift, runtime-instance deltas and heap/listener deltas across at least 20 replay/unmount cycles.

## 18. Recommended architecture

1. Introduce a versioned `SceneContract`: event/action trigger, payload schema, host/instance ID, required/optional targets with visible/unique rules, libraries/property ownership, modes, labels/audio, interruption priority, timeout, final-state policy, replay policy and acknowledgment policy.
2. Mount one persistent Player `ProgressionSceneHost` outside section presence. It owns global event presentation and can hand settled state to a section; section components never determine whether a mandatory event can be presented.
3. Make `AnimationDirector.play()` return a structured receipt: target counts, visibility, duplicate/ownership errors, labels reached, mode, duration, interruption, fallback and semantic final state. Mandatory scenes cannot acknowledge viewed until the receipt meets policy or the user explicitly skips.
4. Persist replayable event payloads and presentation receipts. Replay never repeats a mutation and survives refresh.
5. Enforce wrapper-level ownership. Motion outer wrappers handle UI/layout; GSAP/Rive/Lottie/CSS child nodes handle their declared properties; StPageFlip owns curl host nodes. Strip scene ownership attributes from hidden PageFlip sources or exclude them by host/instance.
6. Resolve motion once at the provider and distribute it to every engine. Add project-authored Rive assets behind validated contracts and retain explicit static fallbacks.
