# Project One Voyage architecture inventory

## Baseline and method

This inventory was taken from the `integration/lanternwake-phase5-universal-language` checkout at `a0a2111c` before the Project One Voyage cutover. It uses Prisma schema inspection, imports, Next route registration, API clients, direct database calls, seed tooling, tests, and documentation references. `Campaign` is the legacy Companion aggregate. `Chronicle` is the renamed canonical authored aggregate (called `Chronicle` in the baseline schema). No conclusion of “unused” is made from a missing UI link alone.

Reachable legacy production surfaces at the baseline are `/tale/[campaignSlug]`, `/quartermaster`, `/api/player/access`, `/api/player/[campaignSlug]/*`, and `/api/gm/*`. They read and, in several cases, write the legacy tables. Canonical surfaces are `/studio`, `/tales`, `/play/*`, `/player/*`, `/captain/*`, `/join/[token]`, `/api/studio/*`, `/api/tales/*`, `/api/play/*`, `/api/player/playthroughs/*`, `/api/captain/*`, and `/api/invitations/*`.

## Cutover conclusion

At the baseline, two independent authoritative writers exist. The legacy writer updates Campaign-derived tables and its own event/snapshot/audit records; the newer writer updates Chronicle Session records and its own events/audit records. Project One Voyage must route the legacy APIs through one canonical command, projection, policy, and audit path before any legacy tables are made read-only. Chronicle is the sole active product/domain term in live code, active documentation, types, and paths; only forward database compatibility SQL may name a predecessor table when required to preserve installed data.
